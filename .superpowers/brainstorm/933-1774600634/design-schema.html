Section 3 of 3: KV storage layout, API routes, cron schedule, and platform adapter pattern.

KV Storage Keys

// Content Kit — one per article
social:kit:{articleId} → ContentKit object
social:kits:index → list of articleIds with kits

// Schedule queue — sorted by fire time
social:queue → sorted set: score=epochMs, member=postRef
social:postref:{id} → { kitId, platform, scheduledAt, status }

// Posted history
social:posted:index → list of postRef IDs (newest first)

// Platform credentials (already exists in localStorage)
→ keep in localStorage, not KV (client-side auth tokens)

API Routes — single catch-all api/social/[...slug].js

Generation
POST /social/generate
→ LLM all platforms + image + video in parallel
→ stores kit in KV, returns kitId
Kit CRUD
GET /social/kits
GET /social/kits/:id
PATCH /social/kits/:id
→ edit text, toggle approved, set scheduledAt
Distribution
POST /social/post
→ fires one platform post immediately
→ called by cron or manual "Post Now"
Schedule
GET /social/schedule
POST /social/deploy
→ deploy writes approved platforms to queue
→ cron reads queue every 5 min

Platform Adapter Pattern

lib/social/platforms/
├── instagram.js ← Graph API: image POST, video Reel upload
├── tiktok.js ← Content Posting API: video upload
├── linkedin.js ← Posts API: text + image (UGC Post)
├── twitter.js ← v2 API: tweet thread (3 posts chained)
├── facebook.js ← Graph API: Page post text + photo
└── index.js ← post(platform, kit) dispatcher

lib/social/media.js ← OpenRouter images + FAL.ai video
lib/social/ava-prompts.js ← hooks, CTAs, pillar system prompts
lib/social/scheduler.js ← auto-schedule across 7 days
